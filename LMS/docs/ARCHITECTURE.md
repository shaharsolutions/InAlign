# ארכיטקטורת מערכת LMS ארגונית

## 1. אפיון מבנה המערכת (System Overview)
המערכת מיועדת לשוק הארגוני ומאפשרת ניהול למידה המבוסס על חבילות SCORM. המערכת נבנית בארכיטקטורה מודולרית של Single Page Application (SPA) תוך שימוש ב-Vanilla JavaScript, HTML ו-CSS מודרני, על גבי שכבת שירותי Backend שיינתנו על ידי Supabase.
המטרה היא לייצר מערכת בעלת מראה "נקי", מקצועי ורספונסיבי, עם תמיכה מלאה בתצוגת RTL ושפה העברית.
הארכיטקטורה מחולקת לשני אזורים נפרדים לחלוטין:
- **צד לומד (Learner Portal):** חיפוש לומדות, צפייה בסטטוס התקדמות והפעלת נגן SCORM לצורך למידה.
- **צד מנהל (Admin Portal):** העלאת לומדות (SCORM), ניהול משתמשים, מעקב התקדמות ודוחות.

## 2. ארכיטקטורת מסכים (Pages Architecture)
### מסכים משותפים:
1. **מסך התחברות (Login):** שדה שם משתמש (אימייל) וסיסמה, עם מנגנון שכחתי סיסמה (בעתיד). מתבסס על Supabase Auth.
### מסכי בצד הלומד (Learner):
1. **דשבורד לומד (My Learning):** הצגת לומדות שמשויכות אליי. פירוט סטטוס (טרם הותחל, בתהליך, הושלם), ציון ובר התקדמות.
2. **נגן הקורס (Course Player):** מסך ייעודי הכולל iFrame לטעינת לומדת SCORM. המסך מאזין לאירועי SCORM מה-iFrame ומדווח אותם ל-Backend.
### מסכי בצד המנהל (Admin):
1. **דשבורד מנהל (Admin Dashboard):** נתונים כלליים – כמה לומדים פעילים, כמה לומדות קיימות.
2. **ניהול לומדות (Courses Management):**
   - רשימת לומדות במערכת וסטטוס (מפורסם/טיוטה).
   - פעולות: העלאת חבילת SCORM חדשה (ZIP), עריכת פרטים (שם, תיאור, קטגוריה).
3. **ניהול משתמשים (User Management):** טבלת משתמשים מבוססת תפקידים, הוספת לומד בודד או הגדרתו כמנהל (על ידי Super Admin).
4. **מסך מעקב ודוחות (Reports & Tracking):**
   - טבלת מעקב לפי משתמש או לומדה (סטטוס, ציון, זמן למידה, תאריכי התחלה וסיום).
   - אופציות סינון מתקדמות וכפתורי ייצוא (Excel, CSV, PDF).

## 3. מבנה בסיס נתונים ב-Supabase (Database Schema)
להלן המבנה המומלץ והקשרים בניהם:

* **users** (מנוהל אוטומטית ע"י Supabase ב-`auth.users`)
* **profiles** (פרופיל משתמש מצורף לכל auth.user)
  - `id` (UUID, PK, רפרנס ל-auth.users.id)
  - `full_name` (Text)
  - `role` (String - `admin` / `learner`)
  - `created_at` (Timestamp)
* **courses** (ניהול הרשומות של הלומדות לפני החלוקות לגרסאות נפרדות ב-SCORM)
  - `id` (UUID, PK)
  - `title` (Text)
  - `description` (Text)
  - `category` (Text)
  - `status` (String - `published`, `draft`)
  - `scorm_path` (Text - נתיב באחסון של Supabase Storage)
  - `created_at` (Timestamp)
  - `created_by` (UUID, רפרנס ל-profiles)
* **enrollments** (שיוך משתמש ללומדה)
  - `id` (UUID, PK)
  - `user_id` (UUID, רפרנס ל-profiles)
  - `course_id` (UUID, רפרנס ל-courses)
  - `assigned_at` (Timestamp)
* **learning_attempts / progress_records** (רשומות מעקב של לומד על לומדה מסוימת)
  - `id` (UUID, PK)
  - `enrollment_id` (UUID)
  - `status` (String - `not_started`, `in_progress`, `completed`)
  - `progress_percent` (Integer - 0-100)
  - `score` (Integer - אופציונלי)
  - `time_spent_seconds` (Integer)
  - `suspend_data` (Text - לשמירת מעקב פנימי של ה-SCORM)
  - `started_at` (Timestamp)
  - `completed_at` (Timestamp)
  - `last_accessed` (Timestamp)

## 4. פירוט הרשאות משתמשים (Row Level Security)
יש ליישם מדיניות RLS ב-Supabase להבטחת אבטחה מירבית:
- `profiles`: כל משתמש רשאי לראות ולערוך את הפרופיל שלו. מנהל (`role = 'admin'`) מורשה לקרוא את כולם.
- `courses`: לומד רשאי לראות רק `status = 'published'`, מנהל רואה ועורך הכול (CRUD מלא).
- `enrollments`: לומד רשאי לראות רק את ההקצאות שלו (`user_id = auth.uid()`). מנהל רואה הכל.
- `progress_records`: לומד מורשה ליצור (Insert) ולעדכן (Update) רק את ההתקדמות שלו עצמו. מנהל מורשה לקרוא הכל (Select) לטובת דוחות, ללא הרשאת עריכה (כדי לשמור על אמינות נתוני ההדרכה).

## 5. פירוט זרימת העלאת SCORM והצגת הלומדה
### תהליך העלאה (מנהל):
1. המנהל בוחר טופס הוספת לומדה, מזין מטא-דאטה (שם, תיאור) ובוחר קובץ ZIP של ה-SCORM (גרסה 1.2 פופולרית מאד במערכות).
2. ה-Frontend שולח את ה-ZIP לאחסון של Supabase (לתוך ה-Bucket "scorm_packages").
3. בעתיד: Edge Function בסופאבייס (או שרת ביניים) נכנס לפעולה, פותח את קובץ ה-ZIP, מזהה את ה-`imsmanifest.xml` ומוצא את ה-entry point (בדרך כלל `index.html` או מספר השקות פנימיות).
4. נוצרת רשומה בטבלת `courses` המצביעה על הנתיב של הלומדה באחסון וסטטוס "פעיל" להצגה ללומדים.

### הצגת הלומדה (לומד):
1. הלומד לוחץ "התחל" בדשבורד.
2. המערכת מייצרת במאגר רשומה של `progress_record` או מביאה אחת קיימת.
3. דף "Course Player" נפתח וטוען iFrame ליעד של קובץ הפתיחה מתוך התיקייה שנפרסה ב-Storage.
4. המערכת כוללת SCORM API Wrapper בצד חלון ה-Parent שמאזין לקריאות ה-JS (`LMSInitialize`, `LMSCommit`, `LMSFinish`, `LMSSetValue` וכו') מתוך הלומדה ומתרגם אותן לקריאות API של Supabase לעדכון ה-`progress_record` (ציון, עמוד אחרון, סטטוס סיום).

## 6. מסכי ניהול ודוחות והמלצה לייצוא
פאנל הניהול ארגוני ואוסף את הנתונים מטבלת ה-`progress_records` (עם JOIN ל-`users` ו-`courses`).
- הנתונים יוצגו בטבלה חכמה עם אפשרות סינון לפי "כלל הארגון" או חיפוש טקסטואלי של עובד ספציפי / קורס ספציפי.
- קיימים פילטרים לפי כל סטטוסי המערכת וטווח תאריכי למידה.

**המלצת ייצוא נתונים ארגוני:**
* **אקסל (Excel - xlsx):** השוק הארגוני מעדיף Excel לייצוא רוב הדוחות לצורך עיבוד במחלקות HR. עדיף להשתמש בספרייה כגון `SheetJS (xlsx)` אשר מסוגלת ליצור קובץ מסודר ונקי ישירות בדפדפן הלקוח.
* **CSV:** לקבצים גדולים וגולמיים ולשילוב המידע אל מול מערכות BI או HRMS חיצוניות בארגון.
* **PDF:** מיועד בעיקר להפקת אישורים/תעודות סיום ספציפיים לכל לומד (באמצעות ספריות כגון `jsPDF` או `pdfmake`).

## 7. תשתית צד הלקוח (Client Frontend)
האפליקציה כתובה תוך שימוש ביכולות הדפדפן (ES Modules) ונבנתה כ-SPA על יד Vanilla JavaScript מודרני מנוהל באמצעות Vite, ללא תלות בספריות צד שלישי כבדות כמו React/Angular כברירת מחדל – אך עם מעטפת שיכולה לעבור בהמשך אינטגרציות נוספות.
עיצוב באוריינטציה מלאה של עברית ו-RTL מימין לשמאל בשילוב עם תמיכה במסכים של מכשירים שונים (רספונסיביות). העיצוב בנוי על סקלת צבעים "קרירה" המיועדת להשראת אמון, אסתטיקה וריכוז לומד (כחול-ארגוני ואפור).
