# סקירת אבטחת מידע - מערכת LMS (InAlign)

מסמך זה מרכז את ממצאי סקירת אבטחת המידע שבוצעה עבור מערכת ה-LMS. הסקירה מתמקדת במנגנוני ההגנה הקיימים, ארכיטקטורת המערכת והמלצות לחיזוק רמת האבטחה.

---

## 1. ניהול זהויות ואימות (Authentication)
המערכת מתבססת על **Supabase Auth** לניהול משתמשים ואימותם.
- **פרוטוקול:** שימוש ב-JWT (JSON Web Tokens) מאובטחים לניהול סשנים.
- **אבטחת סיסמאות:** הסיסמאות אינן נשמרות במסד המערכת ישירות, אלא מנוהלות בצורה מוצפנת על ידי Supabase (BCrypt/Argon2).
- **אימות בצד הלקוח:** קיים מנגנון Guard ב-Router שמוודא שרק משתמשים מחוברים יכולים לגשת לעמודים מוגנים.

## 2. מנגנון הרשאות (Authorization)
המערכת משתמשת במודל **RBAC (Role-Based Access Control)** המשולב עם **RLS (Row Level Security)** ברמת מסד הנתונים.

### תפקידים מוגדרים:
*   **Super Admin:** גישה מלאה לכל הארגונים, הקורסים והגדרות המערכת.
*   **Org Admin:** ניהול משתמשים, קבוצות ותכנים בתוך הארגון שלו בלבד.
*   **Learner:** צפייה בקורסים שהוקצו לו ועדכון התקדמות אישית.

### Row Level Security (RLS):
אבטחת המידע נאכפת ישירות ברמת הטבלאות (PostgreSQL). גם אם משתמש זדוני ינסה לבצע קריאת API ישירה, מסד הנתונים יחסום גישה לנתונים שאינם שלו.
- **בידוד ארגוני (Multi-Tenancy):** שאילתות RLS מוודאות שכל משתמש רואה רק נתונים המשויכים ל-`org_id` שלו.
- **הגנת פרופילים:** משתמש יכול לערוך רק את הפרופיל שלו. מנהלי ארגון יכולים לראות רק את חברי הארגון שלהם.

## 3. אבטחת נתונים ותכנים
- **הצפנה במעבר:** כל התקשורת מתבצעת מעל HTTPS (TLS 1.2+).
- **אחסון קבצים:** קבצי הלומדות (SCORM) מאוחסנים ב-Supabase Storage עם מדיניות אבטחה המאפשרת גישה רק למשתמשים מורשים מאותו ארגון.
- **בידוד לומדות:** הלומדות נטענות בתוך Iframe, מה שמספק רמת בידוד בסיסית מפני סקריפטים זדוניים שעלולים להיות בתוך תוכן הלומדה.

## 4. מנגנוני הגנה בצד הלקוח (Frontend)
- **הגנת נתיבים (Route Guards):** ה-Router מוודא התאמה בין תפקיד המשתמש ליעד הגלישה.
- **מניעת הזרקת קוד (XSS Prevention):** השימוש ב-Frameworks מודרניים וב-Vanilla JS עם Sanitization מונע לרוב הזרקות HTML זדוניות.
- **ממשק ניהול:** כפתורי ניהול ופונקציות רגישות מוסתרים ברמת ה-UI למשתמשים ללא הרשאות מתאימות.

## 5. סיכום נקודות חוזקה
1. **Defense in Depth:** אכיפת אבטחה גם ב-Frontend וגם ב-Database (RLS).
2. **Multi-Tenancy:** הפרדה מוחלטת של נתונים בין ארגונים שונים (Data Isolation).
3. **Audit Trails:** (קיים חלקית) המערכת עוקבת אחרי התקדמות לומדים וזמני גישה.

## 6. המלצות לשיפור (Recommendations)
1. **אימות דו-שלבי (2FA):** מומלץ לאפשר 2FA למשתמשי Super Admin ו-Org Admin דרך Supabase.
2. **מדיניות סיסמאות:** אכיפת סיסמאות מורכבות יותר ושינוי תקופתי (ניתן להגדרה ב-Supabase Dashboard).
3. **Content Security Policy (CSP):** הגדרת כותרות CSP בשרת כדי למנוע טעינת סקריפטים ממקורות לא מורשים.
4. **Rate Limiting:** הגבלת כמות הבקשות ל-API כדי למנוע ניסיונות Brute Force או DOS (מבוצע לרוב ע"י Supabase, אך כדאי לוודא הגדרות).
5. **בדיקות תקופתיות:** ביצוע בדיקת חדירות (Pentest) ייעודית לרכיב טעינת ה-SCORM, שכן הוא מאפשר הרצת קוד צד לקוח.

---
**סטטוס נוכחי:** המערכת עומדת בתקני אבטחה מודרניים עבור אפליקציות SaaS ומספקת הגנה חזקה על פרטיות המשתמשים והמידע הארגוני.
